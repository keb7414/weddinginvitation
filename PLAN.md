# 모바일 청첩장 구현 계획서

> 참고 페이지: https://feelcard.co.kr/channbyeolweddingcard
> 작성일: 2026-06-18
> 목표: 위 모바일 청첩장 페이지의 구조/기능/디자인을 동일하게 구현
>
> **확정 사항**
> - 데이터 저장: **풀스택**(방명록·참석여부·알림신청 DB 저장)
> - 콘텐츠 자산: **아직 없음 → 더미 데이터/플레이스홀더로 먼저 구현 후 교체**

---

## 1. 프로젝트 개요

- **유형**: 모바일 우선(Mobile-First) 단일 페이지 청첩장 (세로 스크롤형)
- **핵심 컨셉**: 신랑(정승찬) · 신부(김은별) 결혼식 안내
  - 일시: 2026년 10월 31일(토) 오후 1시
  - 장소: 경향교회
- **사용 환경**: 카카오톡/문자로 공유되는 링크 → 모바일 브라우저에서 열람
- **반응형**: 모바일 기준(최대 너비 약 420~480px 중앙 정렬), 데스크탑에서도 모바일 폭으로 표시

---

## 2. 기술 스택 제안

| 구분 | 선택지 A (정적/추천) | 선택지 B (풀스택) |
|------|---------------------|------------------|
| 프론트 | Next.js(App Router) + TypeScript | 동일 |
| 스타일 | Tailwind CSS + CSS 애니메이션 | 동일 |
| 백엔드 | 불필요 (정적 배포) | **방명록/참석여부/알림 저장 API 필요 ← 채택** |
| DB | 없음 | **bkend.ai BaaS 또는 별도 DB ← 채택** |
| 지도 | Kakao Map JS SDK | 동일 |
| 공유 | Kakao SDK (JavaScript Key) | 동일 |
| 배포 | Vercel / Netlify | Vercel + 백엔드 |

> **확정: 선택지 B(풀스택)** — 방명록·참석여부·알림신청 데이터 저장을 위해 백엔드/DB 사용.
> 백엔드는 사용자의 기존 환경(Spring) 또는 bkend.ai BaaS 중 택일(아래 6장 참고).

## 3. 페이지 섹션 구성 (위 → 아래 순서)

참고 페이지를 그대로 따라 다음 순서로 구현한다.
※ 원본 상단의 "미리보기용 청첩장입니다" 워터마크/미리보기 배너 프레임은 **구현 대상에서 제외**한다.

1. **메인(커버) 화면**
   - 대표 사진, 신랑·신부 이름, 예식일
   - "Scroll Please" 안내, 배경음악 재생 토글 아이콘
2. **인사말 (INVITATION)**
   - 성구/문구, 초대 메시지
3. **혼주 소개**
   - 양가 부모님 성함 + 장식 구분선
4. **연락하기 (아코디언)**
   - 신랑/신부 및 양가 부모 — 전화/문자 버튼
5. **캘린더 & 갤러리**
   - 예식일 표시 달력 위젯
   - 사진 갤러리(썸네일 그리드 + 더보기 + 라이트박스/슬라이드)
6. **동영상 섹션** (임베드)
7. **오시는 길**
   - 주소, 카카오맵, 지하철/버스/자가용 안내, 주차 정보
8. **신랑·신부 프로필**
9. **인터뷰 (펼침형)**
10. **공지사항 (아코디언)**
11. **마음 전하기 (계좌안내)**
    - 계좌 복사 버튼, 카카오페이/카드 송금 링크
12. **방명록** (신랑/신부만 열람 가능 옵션)
13. **화환 보내기 예약**
14. **참석 의사 전달 폼**
    - 참석 여부 / 식사 여부 / 셔틀버스 이용 체크
15. **예식 알림 신청** (알림 시점 선택)
16. **공유하기**
    - 카카오톡 공유, URL 복사
17. **푸터** (저작권/브랜딩)

## 5. 디자인 / 인터랙션 가이드

- **톤**: 화이트·아이보리 배경, 세리프 계열 국문 폰트(예: 나눔명조/Noto Serif KR), 포인트 색은 차분한 톤
- **애니메이션**: 스크롤 진입 시 페이드/슬라이드업(Intersection Observer)
- **인터랙션 요소**:
  - 아코디언(연락처/공지)
  - 갤러리 라이트박스 + 스와이프
  - 계좌/URL 복사 → 토스트 알림
  - BGM 재생/정지 토글
  - 카카오맵 마커 + 길찾기 버튼
- **접근성**: 버튼 영역 충분히 크게(터치 44px+), 이미지 alt 제공

## 6. 결정 필요 사항 (구현 전 확인)

- [x] **데이터 저장 여부**: → **저장함(풀스택)** 확정
- [x] **콘텐츠 자산**: → **아직 없음, 더미로 진행** 확정
- [ ] **백엔드 선택**: 기존 Spring 프로젝트 활용 vs bkend.ai BaaS vs Next.js Route Handler+DB
- [ ] **외부 키**: Kakao JavaScript Key(지도·공유) 발급 여부
- [ ] **배포 도메인**: 커스텀 도메인 사용 여부

### 6-1. 저장 데이터 모델 (풀스택 확정에 따른 초안)

| 테이블 | 주요 컬럼 |
|--------|-----------|
| `guestbook` | id, name, message, password(삭제용), created_at |
| `rsvp` | id, side(신랑/신부), name, attend(Y/N), headcount, meal(Y/N), shuttle(Y/N), created_at |
| `alarm` | id, contact, notify_at(알림 시점), created_at |

> 방명록은 비밀번호 기반 본인 삭제, 관리자(신랑/신부) 열람 옵션 고려.

## 7. 구현 단계(마일스톤)

| 단계 | 작업 | 산출물 |
|------|------|--------|
| 1 | 프로젝트 셋업(Next.js + Tailwind) | 빈 페이지 빌드 |
| 2 | 정적 섹션 구현(커버~푸터, **더미 콘텐츠/플레이스홀더**) | 스크롤 가능한 전체 골격 |
| 3 | 인터랙션(아코디언/갤러리/복사/BGM) | 동작하는 UI |
| 4 | 외부연동(카카오맵·공유) | 지도·공유 동작 |
| 5 | **백엔드 연동(방명록·RSVP·알림 저장)** | 데이터 저장 동작 |
| 6 | OG 메타·SEO·성능 최적화 | 공유 미리보기 |
| 7 | 배포 | 운영 URL |

## 8. 참고/주의

- 원본 사이트의 **이미지·문구·음원 등 저작 자산은 그대로 복제하지 않고**, 구조/레이아웃/기능을 동일하게 구현하며 실제 콘텐츠는 새로 제공받아 적용한다.
- 모바일 청첩장 특성상 **공유 미리보기(OG 태그)** 와 **로딩 속도**가 사용자 경험에 가장 중요하다.
